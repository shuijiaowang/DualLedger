# DualLedger 数据库设计（AI 开发参考版）

> 本文件是给 AI 协作/开发使用的详细规范。开发时请以本文件为准。
> 用户通俗版见 `doc/开发者我自己的文档-仅可读/数据库设计-通俗版.md`，二者表结构保持一致。
>
> 技术栈：MySQL 8.0（本地）/ 5.7（线上兼容）、Go + GORM、字符集 `utf8mb4_unicode_ci`。

---

## 1. 设计原则

1. **双视图解耦**：一次业务 = 一张 `transaction` 主单 + N 条 `cash_entry`（现金流视图）+ N 条 `accrual_entry`（权责视图）。两种视图互不干扰，各自可做统计。
2. **固定列 + JSON 混合**：固定且会参与查询/索引的用正常列；规则参数、用户自定义扩展用 JSON 列（`amortize_rule`、`ext`）。避免 EAV。
3. **软删除**：所有业务表都嵌入 `gorm.Model`（含 `created_at / updated_at / deleted_at`）。查询默认过滤软删除。
4. **多租户轻量化**：一切业务表都必须带 `user_id` 且建索引；暂不做真正多用户协作，"女友账户/家庭账户"统一作为虚拟账户存在于当前用户下。**预留**`owner_user_id`、`payer_user_id`，但 MVP 阶段固定 = 当前 `user_id`。
5. **金额**：统一使用 `DECIMAL(14,2)`，单位元。不要用 float/double。禁止用"分"做整数存储（前端处理成本更高）。
6. **时间**：统一 `DATETIME`（不是 TIMESTAMP，避开 2038 与时区转换坑）。业务发生时间字段以 `*_at` 命名。
7. **命名**：下划线小写，表名单数（`user`、`account`），符合 GORM 默认规则。
8. **索引**：高频查询围绕 `(user_id, *_at)` 组合；外键约束在 MVP 阶段**不加**（降低迁移摩擦），由应用层保证。
9. **账户哲学**：`account` 代表"一个独立核算单元"（独立的钱袋/小账本），**不等于**支付方式。新用户默认只有 1 个"主账户"；需要细分微信/支付宝/卡，或开辟"女友代付/育儿/彩礼金"等专项账本时再添加。详见 §5。
10. **分类与标签初始化**：系统预设一份基础分类/标签模板（代码静态常量），新用户首次登录时拷贝到其 `user_id` 下；之后用户完全拥有自己的副本，增删互不干扰。详见 §5。
11. **标签双向打法**：`tag` 同时支持打在 `transaction`（购买视角）和 `accrual_entry`（消耗视角）上；生成 accrual_entry 时**默认继承**其 transaction 的标签，可被用户按需增删。这是"一年吃了多少鸡蛋"能准确统计的关键。

## 2. 核心模型关系（ER 概览）

```
user ──┬─< account                     （钱袋子 / 独立核算单元）
       ├─< category                    （分类，树，用户自有副本）
       ├─< tag                         （标签，用户自有副本）
       ├─< resource                    （可摊销资源定义）
       └─< transaction                 （交易主单）
              │
              ├─< cash_entry           （现金流条目：钱何时动）
              ├─< accrual_entry ──< accrual_entry_tag ─> tag （消耗视角）
              └─< transaction_tag ─> tag                    （购买视角）

resource ──< accrual_entry             （资源的每次消耗）
category ──< transaction / accrual_entry
account  ──< cash_entry
```

> 关键业务不变式：
> - `transaction.amount` = `Σ cash_entry.amount`（当 type=TRANSFER 时为单侧，见 §4.5）
> - 若关联了 `resource`，则 `Σ accrual_entry.amount` 最终应等于 `resource.total_cost`（允许期间浮动，结算完成时精确闭合）
> - 若交易的 accrual 是"立即一次性"的（如一日三餐），则 `Σ accrual_entry.amount == transaction.amount`

## 3. 表结构

所有表统一追加：如gorm.Model
```
id          BIGINT UNSIGNED PK AUTO_INCREMENT
created_at  DATETIME NOT NULL
updated_at  DATETIME NOT NULL
deleted_at  DATETIME NULL   INDEX
```
下文字段列表中**省略**上述公共字段。

### 3.1 `user` — 用户

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| username | VARCHAR(64) | 否 | 唯一；登录名 |
| password_hash | VARCHAR(128) | 否 | bcrypt |
| nickname | VARCHAR(64) | 是 | 显示名 |
| avatar | VARCHAR(255) | 是 | 头像 URL |

**索引**：`UNIQUE(username)`

---

### 3.2 `account` — 资金账户

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | 所属用户 |
| name | VARCHAR(64) | 否 | 显示名：如"招商银行"、"女友的钱" |
| type | VARCHAR(16) | 否 | 见枚举 `AccountType` |
| balance | DECIMAL(14,2) | 否 | 当前余额（由应用层维护；冷备可 rebuild） |
| currency | CHAR(3) | 否 | 默认 `CNY` |
| is_virtual | TINYINT(1) | 否 | 1=虚拟账户（代付：女友/家人/共同） |
| is_archived | TINYINT(1) | 否 | 1=已归档，不再出现在录入默认列表 |
| sort | INT | 否 | 排序 |
| note | VARCHAR(255) | 是 | 备注 |

**`AccountType` 枚举**：`CASH`（现金）/ `WECHAT`（微信）/ `ALIPAY`（支付宝）/ `BANK`（银行卡）/ `CREDIT`（信用卡）/ `VIRTUAL`（虚拟/代付）/ `OTHER`

**索引**：`(user_id, is_archived, sort)`

> 设计要点：
> - "女友账户"= `type=VIRTUAL, is_virtual=1`。女友给我钱帮她点外卖：女友虚拟账户→我真实账户的 TRANSFER（现金流），外卖消费走这个虚拟账户支出（权责）。
> - `balance` 为应用层维护的冗余字段；提供"根据 cash_entry 重算"的后台任务/接口。

---

### 3.3 `category` — 分类（树形）

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| parent_id | BIGINT UNSIGNED | 是 | NULL 表示一级分类 |
| name | VARCHAR(64) | 否 | |
| kind | VARCHAR(16) | 否 | `INCOME` / `EXPENSE` / `TRANSFER` / `OTHER` |
| icon | VARCHAR(64) | 是 | 图标名 |
| color | VARCHAR(16) | 是 | #RRGGBB |
| sort | INT | 否 | |

**索引**：`(user_id, kind, parent_id, sort)`

> 设计要点：
> - 一级示例：餐饮、交通、娱乐、工资、会员；二级示例：餐饮>早餐、餐饮>外卖。
> - 统计"我一年吃了多少鸡蛋"MVP 用 tag，不用 category。

---

### 3.4 `tag` — 标签

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| name | VARCHAR(64) | 否 | |
| color | VARCHAR(16) | 是 | |

**索引**：`UNIQUE(user_id, name)`

---

### 3.5 `transaction_tag` — 交易-标签多对多（购买视角）

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| transaction_id | BIGINT UNSIGNED | 否 | |
| tag_id | BIGINT UNSIGNED | 否 | |

**索引**：`UNIQUE(transaction_id, tag_id)`；`(tag_id)` 反查。

> 语义：**这笔交易购买/涉及了什么**。例如外卖 transaction 上打 `#鸡蛋`、`#米饭`；买蛋白粉 transaction 上打 `#蛋白粉`、`#健身`。

---

### 3.5b `accrual_entry_tag` — 权责-标签多对多（消耗视角）

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| accrual_entry_id | BIGINT UNSIGNED | 否 | |
| tag_id | BIGINT UNSIGNED | 否 | |

**索引**：`UNIQUE(accrual_entry_id, tag_id)`；`(tag_id)` 反查。

> 语义：**这一次消耗了什么**。适合"一年吃了多少鸡蛋"这种消耗量统计。
>
> **默认继承规则**：生成 `accrual_entry` 时，应用层自动把其 `transaction_id` 对应的 `transaction_tag` 批量插入到此表；用户在打卡/手动消耗时可以追加或删除标签，不回写到 transaction_tag。
>
> 外卖场景（购买=消耗同一天）：打一次 transaction 标签就自动传递到 accrual_entry。
>
> 跨天消费场景（一袋鸡蛋 30 个）：transaction 打 `#鸡蛋` 只是元数据，真正统计消耗量应该查 `accrual_entry_tag`（每次吃时 qty 和标签都记录）。

---

### 3.6 `resource` — 可摊销资源定义

> **何时建 resource？** 需要跨多天/多次消耗、或需要按次数/周期摊销的物品才建。一日三餐这种"买完即吃完"无需建。

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| name | VARCHAR(128) | 否 | 电动牙刷 / 一箱苹果 / 年会员 |
| category_id | BIGINT UNSIGNED | 是 | |
| unit | VARCHAR(16) | 是 | "天" / "个" / "次" / "ml"... |
| total_qty | DECIMAL(14,4) | 是 | 总量（6 个、30 天、null） |
| remaining_qty | DECIMAL(14,4) | 是 | 剩余量（应用层维护） |
| total_cost | DECIMAL(14,2) | 否 | 总成本（= 购买金额） |
| amortized_cost | DECIMAL(14,2) | 否 | 已摊销累计（冗余，便于快速查询） |
| amortize_rule | JSON | 否 | 摊销规则，见 §4.2 |
| status | VARCHAR(16) | 否 | 见 `ResourceStatus` |
| purchase_at | DATETIME | 否 | 购买时间（出现金流） |
| start_use_at | DATETIME | 是 | 开始使用时间（摊销起点） |
| end_at | DATETIME | 是 | 结束时间（status 改变时写入） |
| note | VARCHAR(512) | 是 | |
| ext | JSON | 是 | 用户自定义字段（单价、店铺、购买链接、评价...） |

**`ResourceStatus` 枚举**：`ACTIVE`（使用中） / `PAUSED`（暂停：如 Steam 游戏吃灰）/ `ENDED`（正常结束）/ `RETURNED`（退货）/ `DISCARDED`（丢弃/烂了）

**索引**：`(user_id, status, start_use_at)`；生成列 `rule_type` 来自 `amortize_rule->'$.type'` 并加索引，用于"查询所有按天摊销的资源"。

---

### 3.7 `transaction` — 交易主单

一次业务发生的"单据"，派生 0~N 条 `cash_entry` 和 0~N 条 `accrual_entry`。

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| owner_user_id | BIGINT UNSIGNED | 否 | 消费归属人（MVP = user_id，预留给代付） |
| payer_user_id | BIGINT UNSIGNED | 否 | 实际出钱人（MVP = user_id） |
| type | VARCHAR(16) | 否 | 见 `TransactionType` |
| occur_at | DATETIME | 否 | 业务发生时间 |
| amount | DECIMAL(14,2) | 否 | 金额（总额，INCOME/EXPENSE 为正，内部符号由 type 决定） |
| account_id | BIGINT UNSIGNED | 是 | 主账户（TRANSFER 的出方） |
| to_account_id | BIGINT UNSIGNED | 是 | TRANSFER 的入方 |
| category_id | BIGINT UNSIGNED | 是 | |
| resource_id | BIGINT UNSIGNED | 是 | 关联的资源（如买电动牙刷） |
| title | VARCHAR(128) | 是 | 一句话描述 |
| note | VARCHAR(1024) | 是 | |
| ext | JSON | 是 | 自定义字段：店铺、平台、评价、购买链接... |

**`TransactionType` 枚举**：
- `INCOME` 收入（工资、红包）
- `EXPENSE` 支出（购物、吃饭）
- `TRANSFER` 转账（银行卡→微信、女友虚拟账户→我的）
- `LOAN` 借贷（借入/借出，由 account 方向区分）
- `DEPOSIT` 押金（可退）
- `REFUND` 退款/退货（反向，通常关联原 transaction）
- `ADJUST` 调整（余额修正、请假扣款等派生项）

**索引**：
- `(user_id, occur_at)` 主要查询路径
- `(user_id, type, occur_at)` 按类型过滤
- `(resource_id)` 反查资源的全部交易
- `(category_id)`

---

### 3.8 `cash_entry` — 现金流条目（钱何时动）

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| transaction_id | BIGINT UNSIGNED | 否 | |
| account_id | BIGINT UNSIGNED | 否 | |
| direction | VARCHAR(4) | 否 | `IN` / `OUT` |
| amount | DECIMAL(14,2) | 否 | 正数 |
| happen_at | DATETIME | 否 | 现金流发生时间（= transaction.occur_at，或延后如信用卡还款） |
| note | VARCHAR(255) | 是 | |

**索引**：`(user_id, happen_at)`、`(account_id, happen_at)`、`(transaction_id)`

> 产生规则（§4.4 详）：
> - INCOME → 1 条 IN
> - EXPENSE → 1 条 OUT
> - TRANSFER → 1 条 OUT（from） + 1 条 IN（to）
> - REFUND → 反向 1 条 IN

---

### 3.9 `accrual_entry` — 权责条目（价值何时耗）

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| transaction_id | BIGINT UNSIGNED | 否 | |
| resource_id | BIGINT UNSIGNED | 是 | 关联资源；一次性消费（吃饭）为 NULL |
| category_id | BIGINT UNSIGNED | 是 | |
| amount | DECIMAL(14,2) | 否 | 金额（正数为消耗；退货/校正可为负） |
| qty | DECIMAL(14,4) | 是 | 消耗数量（1 个、0.5 瓶） |
| unit | VARCHAR(16) | 是 | |
| accrue_at | DATETIME | 否 | 权责发生时间（当天/该次） |
| source | VARCHAR(16) | 否 | `AUTO`（规则自动生成）/`MANUAL`（手动打卡）/`END_SETTLE`（结束结算一次性摊入）/`ADJUST` |
| note | VARCHAR(255) | 是 | |

**索引**：`(user_id, accrue_at)`、`(resource_id, accrue_at)`、`(transaction_id)`、`(category_id, accrue_at)`

## 4. 关键业务规则

### 4.1 双视图产出规则（生成方向）

用户操作 → 应用层在事务中创建：

| 场景 | transaction | cash_entry | accrual_entry |
|---|---|---|---|
| 吃饭 15 元 | 1 条 EXPENSE | 1 条 OUT | 1 条（accrue_at = occur_at，source=AUTO，立即摊） |
| 买电动牙刷 150 元 | 1 条 EXPENSE + resource_id | 1 条 OUT | **不立即生成**，由摊销引擎按规则生成 |
| 每日摊销定时任务 | 无 | 无 | 为 ACTIVE 资源按 FIXED_PERIOD/SCHEDULED 规则补齐当天条目 |
| 吃 1 个苹果 | 无（不产生交易） | 无 | 1 条（source=MANUAL, resource_id=苹果箱） |
| 资源 ENDED（烂了）| 无 | 无 | 1 条（source=END_SETTLE，金额=剩余成本，可正可负） |
| 工资 4500 | 1 条 INCOME + resource_id(工资包) | 1 条 IN | 按天 30 条 AUTO |
| 请假 1 天 | 1 条 ADJUST（负向）| 可选：1 条 OUT（实际不扣现金） | 1 条（负值，抵消当天工资摊销） |
| 转账 | 1 条 TRANSFER | 1 OUT + 1 IN | 无 |
| 退货 | 1 条 REFUND（关联原 transaction） | 1 IN | 相应反向 accrual（把已摊的扣回） |

> **标签继承规则（重要）**：上表中每次新建 `accrual_entry` 时，应用层必须在**同一事务内**，把该条目所属 `transaction` 的全部 `transaction_tag` 记录拷贝一份到 `accrual_entry_tag`。MANUAL 打卡补录时，若用户未手动选标签，则继承 transaction；若手动选择了，则以用户选择为准。

### 4.2 `amortize_rule` JSON Schema

```json
// 类型 A：按固定天数平摊
{
  "type": "FIXED_PERIOD",
  "days": 30,
  "start": "2026-04-10",    // 缺省取 resource.start_use_at
  "include_start_gap": false // 是否把购买到使用之间的"空窗天"也分摊
}

// 类型 B：按总次数/总量摊销（一箱苹果 6 个）
{
  "type": "BY_COUNT",
  "total_qty": 6,
  "manual_consume": true    // 吃一个打一次卡，应用层把剩余数量→单次金额
}

// 类型 C：按计划表（蛋白粉周一三五七喝）
{
  "type": "SCHEDULED",
  "weekdays": [1, 3, 5, 7],   // ISO: 1=Mon ... 7=Sun
  "per_use_qty": 1
}

// 类型 D：完全手动打卡（燕麦、可乐）
{
  "type": "MANUAL",
  "per_use_qty": null        // 用户每次打卡自己填消耗比例
}

// 类型 E：动态摊销（电动牙刷：每用一天，当前单日成本 = 剩余成本/剩余预期天数）
{
  "type": "DYNAMIC_BY_DAY",
  "expected_days": 30,       // 预期寿命
  "actual_used_days": 15     // 冗余字段，引擎每日更新
}
```

摊销引擎选型建议：
- 每日 00:30 定时任务扫描 `resource.status=ACTIVE` 的行
- 按 `rule_type` 分发处理器
- 幂等：`(resource_id, accrue_at::date)` 唯一约束，重复运行不会重复计入
- MANUAL / BY_COUNT + manual_consume 不走定时任务，由用户"打卡"接口即时生成条目

### 4.3 账户余额一致性

- 应用层在每条 `cash_entry` 提交时更新对应 `account.balance`（事务内）
- 提供一个"根据 cash_entry 重算账户余额"的后台接口（修数据用）
- `is_virtual` 账户余额允许为负（女友给我钱→我消费 → 虚拟账户余额负数表示还欠她多少，这是正常的）

### 4.4 转账/借钱/押金的处理约定

| 场景 | transaction.type | cash_entry | accrual_entry |
|---|---|---|---|
| 银行卡→微信 100 | TRANSFER | 1 OUT（银行）+ 1 IN（微信） | 无 |
| 女友给我 50 帮买外卖 | TRANSFER | 1 OUT（女友虚拟）+ 1 IN（我微信） | 无 |
| 外卖花 50（女友的） | EXPENSE，owner_user_id=女友虚拟用户/account | 1 OUT（我微信）| 1 条（category=外卖） |
| 借出 200 给朋友 | LOAN | 1 OUT（我）+ 1 IN（"朋友"虚拟账户） | 无 |
| 朋友还钱 200 | LOAN 或 TRANSFER | 1 IN（我）+ 1 OUT（朋友虚拟）| 无 |
| 付押金 500 | DEPOSIT | 1 OUT（我）+ 1 IN（"XX 押金"虚拟账户）| 无 |

> 全部借贷/押金通过"虚拟账户"建模，不进入权责视图，满足"这部分好像不需要权责视图"的直觉。

### 4.5 退货

- 新建 `type=REFUND` 的 transaction，`ext.original_transaction_id` 指向原 transaction
- 生成反向 cash_entry（金额相等，方向相反）
- 若原交易关联了 resource：将 resource.status 置为 `RETURNED`，并生成一条 accrual（负值，金额 = `amortized_cost`，将已摊销的归零；若选择"不倒扣过去"，则只停止未来摊销）
- 具体策略（"倒扣 vs 只停未来"）由前端让用户选，默认"只停未来"，避免报表回溯剧烈变动。

### 4.6 工资与请假的最小方案

1. 入职/发薪时建 `resource`：name="2026年4月工资"，total_cost=4500，amortize_rule=`{type:FIXED_PERIOD,days:30,start:"2026-04-01"}`。
2. 发薪当天建 `transaction type=INCOME, amount=4500, resource_id=工资id`，产生 1 条 IN cash_entry。
3. 定时任务每天产生 1 条 AUTO accrual（+150）。
4. 10 号请假：建 `transaction type=ADJUST, amount=-150, resource_id=工资id`，产生 1 条 -150 accrual（source=ADJUST，accrue_at=10号）。现金流可以不产生条目（工资已发），也可以按公司扣款日生成一个单独 OUT。
5. 15 号迟到扣 20：同 4，amount=-20。
6. 社保扣 450：若在发薪前扣，则 income.amount 直接 4050；若是独立扣款，建独立 EXPENSE。

### 4.7 软删除与审计

- 所有业务表 `deleted_at` 控制软删除，默认查询过滤
- `ext.audit_log` 可选用于保留变更历史（MVP 不做，预留思路）

## 5. 数据初始化与静态模板

### 5.1 账户初始化

- **新用户注册时**，后端仅创建 1 条 `account`：`name="主账户", type=OTHER, is_virtual=0, sort=0`。
- 不预建微信/支付宝/银行卡；用户若需细分，自己在设置里添加。
- 前端"选择账户"组件：当用户只有 1 个账户时**完全隐藏**（自动选中），多于 1 个时才展示选择器。降低普通用户心智负担。

### 5.2 分类/标签静态模板（代码常量 + 初始化拷贝）

**存放位置**：`service/model/presets.go`（Go 常量/变量，编译期即存在）。

**初始化时机**：
- 用户注册成功后，或首次登录 `GET /me` 时检测 `category` / `tag` 表在该 `user_id` 下为空 → 触发拷贝。
- 拷贝后完全属于该用户，用户改名/删除互不影响其他用户。

**好处**：
- **零关联**：表结构不存"模板 id"、"谁创建"；个人数据就是个人数据。
- **离线友好**：应用重新部署/换库，模板随代码走，不依赖数据库的预置行。
- **用户可自由演化**：预设只是起点，用户完全控制自己的分类树。

**预设清单（MVP 精简版，可后续扩展）**：

```

  - 餐饮
      └ 早餐、午餐、晚餐、外卖、零食饮料
  - 交通

      └ 日用品、服饰、电子产品
  - 居家
      └ 房租水电、通讯网络
  - 娱乐
      └ 订阅会员、游戏、出行
  - 医疗健康
  - 其他支出

INCOME（收入）：
  - 工资
  - 奖金
  - 理财
  - 红包/礼金
  - 其他收入
```

**预设标签（轻量，仅给几个常用，多数鼓励用户按需新增）**：
```

```

### 5.3 关于"全局分类池 + 用户偏好 JSON"方案（暂不采用，记录于此）

开发者曾提出一种思路：分类表去掉 `user_id`，做成全局池；用户侧存一份 JSON 记录"用过哪些/偏好顺序"，以此避免"每用户一份拷贝"的重复。

**优点**：全局唯一、数据量小、未来方便做"分类社交共享"。
**缺点（MVP 为何不选）**：
1. 命名碰撞：不同用户叫"餐饮"/"吃饭"/"饮食"，合并逻辑复杂。
2. 隐私：个人自定义分类可能含敏感词。
3. 级联删除/重命名谁说了算？责任不清。
4. 统计查询要多做一层 JOIN / JSON 过滤，性能反而变差。

**MVP 选择**：每用户独立副本 + 代码里静态预设模板。等"用户共享分类市场"这个 V2 功能真正立项时，再做一张 `category_marketplace` 表专门承载，不污染主表。

## 9. 典型查询示例（便于接口开发对齐）

```sql
-- 1. 本月现金流（按日聚合）
SELECT DATE(happen_at) d,
       SUM(CASE direction WHEN 'IN' THEN amount ELSE -amount END) net
FROM cash_entry
WHERE user_id = ? AND happen_at >= ? AND happen_at < ?
  AND deleted_at IS NULL
GROUP BY d;

-- 2. 本月权责（按分类聚合）
SELECT category_id, SUM(amount)
FROM accrual_entry
WHERE user_id = ? AND accrue_at >= ? AND accrue_at < ?
  AND deleted_at IS NULL
GROUP BY category_id;

-- 3. 查询所有按天摊销的活跃资源（利用 JSON 生成列）
SELECT * FROM resource
WHERE user_id = ? AND status = 'ACTIVE'
  AND JSON_EXTRACT(amortize_rule, '$.type') IN ('FIXED_PERIOD','DYNAMIC_BY_DAY');

-- 4. 统计一年吃了多少个鸡蛋（消耗视角，推荐）
--    用 accrual_entry_tag 才能正确计算"实际吃了多少"，
--    并按 qty 求和支持"半个鸡蛋"这种精度
SELECT SUM(ae.qty) AS total_eggs
FROM accrual_entry ae
JOIN accrual_entry_tag aet ON aet.accrual_entry_id = ae.id
JOIN tag g                 ON g.id = aet.tag_id
WHERE ae.user_id = ? AND g.name = '鸡蛋'
  AND ae.accrue_at BETWEEN ? AND ?
  AND ae.deleted_at IS NULL;

-- 5. 统计一年"花了多少钱在鸡蛋上"（购买视角，对偶）
SELECT SUM(t.amount)
FROM transaction t
JOIN transaction_tag tt ON tt.transaction_id = t.id
JOIN tag g              ON g.id = tt.tag_id
WHERE t.user_id = ? AND g.name = '鸡蛋'
  AND t.occur_at BETWEEN ? AND ?
  AND t.deleted_at IS NULL;
```

## 11. 未决与将来

- **用户自定义统计 DSL**：ext JSON 已预留；未来做一个"表达式引擎 + 自定义看板"，不在 MVP。
- **多人协作**：预留 owner_user_id / payer_user_id；将来接真正的多租户需新增 `ledger`（账本）维度。
- **货币与汇率**：currency 已预留，MVP 只支持 CNY。
- **附件/图片**：未来加 `attachment` 表，关联到 transaction / resource。
- **全局分类市场**：如要让用户共享自定义分类，另建 `category_marketplace` 专表，不修改 `category`。见 §5.3。
- **用户使用偏好统计**：未来可给 `user` 加一个 `preferences` JSON 字段，记录"最常用分类/标签排序"，前端用于排序高频项。不影响主表结构。
