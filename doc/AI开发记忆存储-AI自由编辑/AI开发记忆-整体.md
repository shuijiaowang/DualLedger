# AI开发记忆（DualLedger）

## 1. 项目定位
- 项目名称：双记账系统（DualLedger）。
- 核心目标：同时支持两种记账视角，帮助用户既看见“钱何时流动”，也看见“价值何时被消耗”。
  - 现金流视图：记录资金实际收支发生时间。
  - 权责视图：记录消费价值的实际消耗时间，支持分摊/摊销。

## 2. 技术栈理解
- 后端：Go + Gin + MySQL(本地是8.0，服务器上是5.7)（当前目录结构包含 `service/`）。
- PC Web 前端：Vue3 + Vite（当前目录结构包含 `web/`）。
- App/小程序前端：uni-app（当前目录结构包含 `web_uniapp/`，但阶段优先级靠后）。
- 认证与接口方向：已有 JWT、用户注册/登录、示例鉴权接口等基础能力可扩展。

## 3. 当前需求优先级
- 第一优先：后端完善到约 90%（数据库设计、核心接口、业务规则稳定）。
- 第二优先：Web 端测试实用优先，完成可测试的基础页面与流程，目标约 60%（“能看能用”为主，样式可简化）。
- 第三优先：uni-app 在后端与 Web 基本稳定后再迁移/复用实现。

## 4. 任务计划（AI协作版本）
1. 梳理后端领域模型与数据表：如用户、账户、账单、记录、摊销规则等。
2. 先实现并稳定核心 API：认证、记账录入、查询统计、权责摊销计算。
3. 通过 Web 前端快速联调：登录、记账录入、列表查询、基础统计展示。
4. 补齐关键异常处理与权限校验，形成可回归的最小闭环。
5. 在核心流程稳定后，再推进 uni-app 端适配。

## 5. 关键约束与协作原则
- 先“宏观可控”再“快速编码”：开发前先解释思路、阶段目标与影响范围，需要让开发人理解核心逻辑。
- 以业务正确性优先于界面精细度：先保证数据与规则正确，再逐步优化体验。
- 接口与配置尽量一致化：端口、鉴权、错误码、响应结构保持统一，降低前后端联调成本。
- 新功能尽量围绕双视图模型展开，避免偏离“现金流 + 权责”核心价值。
- 每轮迭代保持可运行、可验证，避免一次性大改导致不可控风险。

## 6. 下一步建议（可立即执行）
- 和开发者协作明确需求，设计数据库，
- 明确“权责摊销”规则的最小版本（按天/按次数/固定周期）及输入输出字段。
- 确定记账主流程 API 清单（创建、查询、更新、统计）与字段规范。
- 在 Web 端先做 3 个最小页面：登录页、记账页、记录页，优先打通端到端流程。
