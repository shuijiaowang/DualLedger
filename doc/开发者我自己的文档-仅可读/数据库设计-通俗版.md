## 一句话看懂整个系统

> **一次花钱 = 开一张"单据"，挂两本流水：一本记"钱什么时候动"，一本记"价值什么时候被消耗"。**

这两本流水完全分开，你想看哪本就查哪本，互不干扰。这就是"双记账"。

## 每张表分别干嘛（打比方）

### 1. user 用户
最朴素，就是你的登录账号，没啥好说的。

### 2. account 账户 —— 一个"独立核算单元"
换种说法，就是**一个独立的小账本 / 钱袋**。它**不等于**支付方式。

- **默认只有 1 个"主账户"**。注册后你看到的就是一个主账户，够用的话就用它，别折腾。
- 想细分微信/支付宝/银行卡？自己去设置里加。
- 想给特定事情开专项账本？比如"育儿"、"彩礼金"、"装修基金"、"女友代付"？每个都建一个 account 就行。
- 女友给你 50 帮点外卖：女友账户 → 我的主账户 **转账**一笔（现金流这么走）。具体代付挂谁名下 MVP 先记在你头上，以后升级字段支持（`owner_user_id`/`payer_user_id`已预留）。

> 不是逼你记"这顿饭是微信还是支付宝付的"，是你自己觉得有用再分。

### 3. category 分类 —— 支出长什么样
餐饮 / 交通 / 娱乐 / 工资 / 会员 …… 可以任意深度：饮食 > 水果 > 苹果 > 烟台苹果 都行。

**模板方案**：
- 系统带了一份预设分类模板（写在代码里，不在数据库里写死）
- 你注册时，后端**复制一份**到你自己名下，之后完全属于你，改啊删啊都是你的事，不影响别人。
- 你新建的分类**只属于你**。以后如果想做"全局分类市场/共享"，我们另建表，不污染这张表。

### 4. tag 标签 —— 横切维度
和分类**正交**。例如：一顿外卖分类是"餐饮 > 外卖"，但你可以给它打标签 `#鸡蛋` `#米饭` `#麦当劳`。

标签也跟分类一样：系统给几个预设（`#外卖 #加班 #朋友聚餐 #健身 #出差`），拷贝到你名下，你想加想删随意。

> 想查"我一年吃了多少鸡蛋"？就靠 tag。但注意：**要打在"消耗时"的条目上**才准。见下面第 10 张表。

### 5. resource 资源 —— "一买多用"的东西
**只有需要跨天/跨次使用的才建 resource**，一顿饭不用建。

例子：
- 电动牙刷 150 元（用很多天）
- 年会员 360 元（用 365 天）
- 一箱苹果 18 元（吃 6 次）
- Steam 游戏 68 元（玩几天后丢掉）
- 4 月工资 4500 元（你把工资当成"分 30 天消耗"的"资源"也 OK）

每个 resource 有一条 **摊销规则**（`amortize_rule`），告诉系统"这个东西该怎么算每天消耗多少"。五种规则：

| 规则 | 场景 |
|---|---|
| **按天平摊** `FIXED_PERIOD` | 会员、工资，固定 30 天/365 天平分 |
| **按次数** `BY_COUNT` | 一箱苹果 6 个，吃 1 个 = 18/6 |
| **按计划表** `SCHEDULED` | 蛋白粉一三五七喝，这几天才扣钱 |
| **手动打卡** `MANUAL` | 燕麦、可乐，吃了自己点一下，不吃就不扣 |
| **动态天摊** `DYNAMIC_BY_DAY` | 电动牙刷：用一天算一天，15 天时 10 元/天，30 天时 5 元/天 |

### 6. transaction 交易 —— 一张"单据"
每次"钱动了"或"消费发生"，都开一张 transaction。它会自动派生底下两种流水。

它也有类型：
- `INCOME` 收入（发工资、红包）
- `EXPENSE` 支出（吃饭、买东西）
- `TRANSFER` 转账（卡→微信、女友→你）
- `LOAN` 借出/借入
- `DEPOSIT` 押金
- `REFUND` 退货退款
- `ADJUST` 调整（请假扣工资这种）

### 7. cash_entry 现金流条目 —— "钱何时动"这本账
每一条代表：**哪天、哪个账户、进还是出、多少钱**。
- 买电动牙刷 150：买的那天，1 条 -150（出）
- 转账：1 条出 + 1 条入
- 吃饭：1 条出

### 8. accrual_entry 权责条目 —— "价值何时被消耗"这本账
每一条代表：**哪天、消耗了多少钱的价值**。
- 电动牙刷 150，按天动态摊：系统每天自动生成 1 条，15 天时每天 10 元，第 16 天起变成 9.x 元…
- 一箱苹果：周一吃 1 个生成 1 条（6），周二吃 1 个再生成 1 条，周三吃 2 个生成 2 条，后面烂了丢掉 → 剩余 2 份自动生成 1 条"结束清算"把账平掉。
- 吃饭 15 元：买的那天就生成 1 条 15 元，和现金流同一天（无摊销）
- 工资 4500 分 30 天：系统每天帮你生成 1 条 +150
- 请假 1 天：生成 1 条 -150 抵消当天

### 9. transaction_tag —— **购买视角**的标签桥
一条交易可以打多个标签，一个标签可以贴在多条交易上。这张表就是中间桥。

> 语义："**这笔交易购买/涉及了什么**"。
> 例：外卖订单打 `#鸡蛋` `#米饭`；买蛋白粉打 `#蛋白粉`。

### 10. accrual_entry_tag —— **消耗视角**的标签桥
和 transaction_tag 结构一样，但挂在 accrual_entry 上，代表"**这一次消耗了什么**"。

**为什么要这个？**
- 买一袋鸡蛋 30 个 150 元。如果标签只打在 transaction 上，统计会告诉你"那一天你吃了 30 个鸡蛋"，这不对。
- 正确做法是：买的时候 transaction 可以不打（或者打，无所谓），**每次吃的时候**生成一条 accrual_entry，顺手在上面勾个 `#鸡蛋`，qty=1（或 0.5 个）。
- 年底统计：`SUM(accrual_entry.qty) WHERE tag=鸡蛋` → 这才是真吃了多少。

**会不会很麻烦？** 不会。
- 规则自动生成的 accrual_entry（蛋白粉按计划表、工资按天摊）：**默认继承** transaction 上的标签，一次都不用勾。
- 手动打卡时（吃苹果、喝可乐）：默认也继承，你觉得需要改再改。
- 外卖场景（购买=消耗同一天）：你在 transaction 打的标签自动传下来，等于没多干活。

## 你提过的问题，这套设计怎么答

| 你的原问题 | 答案 |
|---|---|
| 转账/借钱/押金要不要权责视图？ | **不要**。只产生现金流，走 TRANSFER/LOAN/DEPOSIT 类型，借贷双方用"虚拟账户"建模 |
| 女友代付要不要多角色？ | MVP **不搞多用户**。用"女友虚拟账户"解决 80% 需求。字段里预留 `owner_user_id`/`payer_user_id`，将来升级不难 |
| 工资 4500 里社保扣 450 怎么办？ | 两种都行：① 只记 income 4050；② 记 4500 income + 一条 450 的 EXPENSE。推荐 ② 便于统计社保累计 |
| 10 号请假这天不给钱？ | 建一条 `ADJUST` 类型交易，金额 -150，生成 1 条负的权责条目 |
| 电动牙刷 15 天时 10/天，30 天时 5/天？ | `DYNAMIC_BY_DAY` 规则，系统每天都会按"剩余成本 / 剩余预期天数"自动算 |
| 一箱苹果吃不完烂了？ | resource 状态改 `DISCARDED`，系统自动出一条"结束清算"条目把剩余成本一次性算掉（你可以选择"算在扔掉那天"或"分摊到已吃的那几天") |
| 可乐第一天喝三分之一，第二天三分之二？ | `MANUAL` 规则，每次喝自己点一下"消耗 0.33 瓶" |
| 蛋白粉一三五七喝？ | `SCHEDULED` 规则，写 `weekdays:[1,3,5,7]`，系统自动在这几天生成条目 |
| 燕麦吃不固定？ | `MANUAL` 规则，吃了打一下卡，不吃不算 |
| Steam 游戏爽完就不玩了？ | resource 状态改 `PAUSED`（暂停）或 `ENDED`（结束）。改主意了再恢复 `ACTIVE` |
| 键盘下月才到，中间时间算摊销吗？ | 看你：`start_use_at` 设成到货日，中间不算；或 `include_start_gap=true` 把空窗也平摊进去 |
| 退货用了两天？ | 建一条 REFUND 交易，现金流反向入账；权责条目选择"把已摊的倒扣"或"只停未来"，默认只停未来（避免历史报表被改） |
| 买菜单价/数量？ | 交易的 `ext` JSON 里放 `{unit_price, qty, weight}`；需要统计就从 JSON 里提 |
| 想统计"一年吃了多少鸡蛋"？ | 标签打在 **消耗条目（accrual_entry_tag）** 上才准。外卖打一次会自动继承；自己买 30 个吃时每次打卡打一次。带 qty 可以精确到"半个鸡蛋" |
| 消耗时要备注吗？ | `accrual_entry.note` 支持。想统计频次用标签（`#跑步` `#水煮`），想写私人感受用 note |
| 账户要不要全分清楚（微信/支付宝/卡）？ | **默认就 1 个"主账户"**，你懒就别分。想分再去设置加 |
| 像"育儿预算"、"彩礼金"怎么存？ | 也是建一个 account，账户就是"独立核算单元"而不是支付方式 |
| 分类能做到四级吗？ | 能。`parent_id` 自关联，任意深度 |
| 用户自定义分类要共享出去吗？ | **MVP 不共享，每人一份拷贝**，简单可靠。未来想做"分类市场"再另起一张表，不污染现有设计 |
| 评价/备注/购买链接/店铺？ | 交易和资源都有 `note`（短备注）和 `ext` JSON（放链接、店铺名、评分等） |
| null 还是 JSON？ | **两种混合**。会查询/加索引的字段正常列（可 null）；规则参数、用户随意扩展的字段用 JSON。MySQL 8 的 JSON 支持足够 |
| 用户自定义统计？ | `ext` JSON 字段已经预留。做自定义统计 DSL 是 V2 目标，MVP 先不做 |

## 你需要记住的 4 个核心概念

1. **transaction 是单据，cash_entry 和 accrual_entry 是它生成的流水**
   查账看哪本就查哪本表，别混着查。

2. **需要摊销才建 resource，一顿饭不要建**
   建 resource 的"成本"是你要维护规则；不建就是传统一次性记账。

3. **虚拟账户解决一切"不涉及权责"的怪事**
   借钱、押金、女友代付、育儿预算、彩礼金，全部是"另建一个 account"。账户不等于支付方式。

4. **标签两处都能打，消耗视角才是"统计真消耗"的正确打法**
   transaction_tag = 购买视角；accrual_entry_tag = 消耗视角。默认继承，不用多操心。
