> 这份给你自己看的，说人话。AI 开发细节在 `doc/AI开发记忆存储-AI自由编辑/数据库设计.md`。

## 一句话看懂整个系统

> **一次花钱/收钱 = 开一张"单据"（transaction）。钱何时动，直接看这张单据；价值何时被消耗，一部分按规则动态算、一部分由你手动打卡记录。**

"双记账"的意思是：**现金流视图** 和 **权责视图** 两份独立的账，你想看哪本就查哪本。

## v2 相对 v1 做了哪些精简（重要）

你在 v1 上反馈的痛点，v2 都改了：

| 你的反馈 | v2 的改动 |
|---|---|
| "每天都写一条摊销？那数据量不爆炸？房租/会员/电脑天天写太冗余" | **不写**！FIXED_PERIOD / DYNAMIC_BY_DAY 规则用**动态计算**，查询时当场算；只有"请假/打卡/丢弃"这种真实事件才入库 |
| "cash_entry 不就是重复吗，transaction 已经有了" | **删了 cash_entry 表**。单账户直接用 transaction；转账用 `to_account_id` 表达对侧 |
| "代付/借钱要给女友/小王建虚拟账户？太麻烦，我只需要记录出入" | **去掉虚拟账户**。transaction 加个 `counterparty` 文本字段（写"小王"/"女友"），借出 OUT / 还钱 IN 就行 |
| "SCHEDULED 定时任务不合理，能不能默认是算但不存，需要时才存？" | **去掉 SCHEDULED 规则**。蛋白粉一三五七喝直接用 MANUAL 打卡；不打就不算，打就入库 |
| "标签中间表(transaction_tag / accrual_entry_tag)看着麻烦" | **去掉中间表**。tags 直接 JSON 数组写在 transaction 和 accrual_entry 上：`["鸡蛋","外卖"]` |
| "PAUSED 暂停有必要吗？像请假" | **去掉 PAUSED**。买了就一直消耗，想彻底停就 ENDED |
| "分类每用户一份拷贝麻烦，不如系统定义好让用户选" | **改系统级**。MVP 不让用户自定义分类/标签，全部代码常量 |
| "owner_user_id / payer_user_id 不需要，MVP 就一个用户" | **删了** |
| "account.type / is_virtual / color 是不是多余？" | **全删了**。账户只有 name（你愿意叫啥叫啥） |

**表数量**：从 10 张业务表精简到 **5 张**（user / account / resource / transaction / accrual_entry）。

## 5 张表，一图看懂

```
      user（用户）
         │
    ┌────┼──────────────────────┐
    │    │                      │
 account（账户/钱袋）     resource（可摊销资源）
    │                           │
    └─────> transaction <────────┘
               │（一次花钱/收钱/转账 · 自带 tags）
               │
               └─< accrual_entry（权责"真实事件"：打卡/请假/结算）
                    · AUTO 摊销不入此表，查询时动态算
```

系统级常量（不入业务表）：
- `category`：分类树（代码常量，前端拉列表）
- `tag`：标签建议词（代码常量，前端自动补全）

## 每张表分别干嘛（打比方）

### 1. user 用户
就是登录账号，没啥好说的。

### 2. account 账户 —— 一个"独立的小账本 / 钱袋"

- **默认只有 1 个"主账户"**。注册后就一个主账户，够用就用它，不折腾。
- 想细分微信/支付宝/银行卡？自己去设置里加。
- 想开专项账本？比如"育儿预算"、"彩礼金"、"装修基金"？每个都建一个 account 就行。
- 女友给你 50 帮点外卖？**MVP 简记**：当作你的钱花掉，在 transaction 的 `counterparty` 字段里标"女友代付"或打标签 `#女友代付`，想看她那部分就按这个筛。

**字段极简**：`name / balance / is_archived / sort / note`。**没有** `type`，没有 `is_virtual`，也没有 `color`。

### 3. resource 资源 —— "一买多用"的东西

**只有需要跨天/跨次使用的才建 resource**，一顿饭不用建。

例子：
- 电动牙刷 150 元（用很多天）
- 年会员 360 元（用 365 天）
- 一箱苹果 18 元（吃 6 次）
- 4 月工资 4500 元（你也可以把工资当成"分 30 天消耗"的 resource）

每个 resource 有一条 **摊销规则** `amortize_rule`。v2 把规则精简到 **3 种**：

| 规则 | 场景 | 摊销事件是否入库 |
|---|---|---|
| `FIXED_PERIOD` 按天平摊 | 会员、工资、房租，固定 30 天/365 天平分 | **不入库**，查询时动态算每天固定金额 |
| `DYNAMIC_BY_DAY` 动态天摊 | 电动牙刷：用一天算一天，15 天时 10/天，30 天时 5/天 | **不入库**，查询时按"剩余成本/剩余预期天数"动态算 |
| `BY_COUNT` 按次数 | 一箱苹果 6 个 / 蛋白粉 30 次 | **入库**：每次吃打一次卡，写一条 `accrual_entry` |

**"可乐两天喝完 三分之一/三分之二"怎么记？** 用 `BY_COUNT`，`total_qty=1`，每次打卡 qty 填 0.33 / 0.67。

**"燕麦吃不固定"怎么记？** 用 `BY_COUNT`，吃了打一下卡，没吃就不管。

**"蛋白粉一三五七喝"怎么记？** 用 `BY_COUNT`，真喝的那天打一下卡就好。你在 UI 上可以加个"每周一三五七提醒我打卡"的便利功能，但**不往数据库写没真实发生的事**。

**状态** `status`（v2 去掉 PAUSED）：
- `ACTIVE` 使用中
- `ENDED` 正常用完
- `RETURNED` 退货
- `DISCARDED` 丢弃/烂了

### 4. transaction 交易 —— 一张"单据"

每次钱动了或消费发生都开一张 transaction。它同时承担"现金流记录"的职责（v1 里的 cash_entry 已经合并进来了）。

**核心字段**：
- `type` 类型：`INCOME` / `EXPENSE` / `TRANSFER` / `LOAN` / `DEPOSIT` / `REFUND` / `ADJUST`
- `direction` 方向：`IN` / `OUT` / `BOTH`（`BOTH` 专给 TRANSFER 用）
- `amount` 金额（总是正数；方向靠 type + direction 决定）
- `account_id` 主账户（TRANSFER 时是出方）
- `to_account_id` 仅 TRANSFER 用（入方账户）
- `counterparty` **v2 新增**：对手方文本，如"小王"、"房东"、"女友代付"
- `tags` **v2 新增**：JSON 数组 `["鸡蛋","外卖"]`，不再用中间表
- `category_code`、`resource_id`、`title`、`note`、`ext`

**例子**：
- 吃饭 15 元：`type=EXPENSE, direction=OUT, amount=15, account_id=微信, category_code='food.lunch', tags=["鸭腿饭","鸡蛋"]`
- 借给小王 200：`type=LOAN, direction=OUT, amount=200, account_id=微信, counterparty="小王"`
- 小王还 200：`type=LOAN, direction=IN, amount=200, account_id=微信, counterparty="小王"`
- 银行卡转微信 100：`type=TRANSFER, direction=BOTH, amount=100, account_id=招行, to_account_id=微信`

**查"小王欠我多少"**：把所有 `type=LOAN, counterparty='小王'` 的记录求和（OUT 加 IN 减）即可，不用什么虚拟账户 id。

### 5. accrual_entry 权责真实事件 —— "**真正发生的**消耗事件"

**v2 最大变化**：这张表**只存真实事件**，不存规则自动生成的行。

入库的三种事件：
- `MANUAL` 手动打卡：吃 1 个苹果 → 写 1 条
- `ADJUST` 调整：请假扣款 / 迟到扣款 / 退货回冲 → 写 1 条负值
- `END_SETTLE` 结束结算：苹果烂了剩 3 个，丢了 → 写 1 条把剩余 9 元作为损失一次性计入

**不入库的（动态计算）**：
- 电动牙刷每天 5 元 → 不写，查询时算
- 年会员每天 1 元 → 不写，查询时算
- 工资每天 150 元 → 不写，查询时算

**这样做的好处**：
- 数据库不会爆炸。一个 30 天规则的会员，v1 会写 30 行；v2 写 0 行。
- 历史查询照样精确。
- 只要 resource 的 `amortize_rule` 和 `total_cost` 没改，任何时间点查询都得到相同结果。

**担心什么**：查询要做动态计算。但"按月/按日聚合"这种粒度量级很小（几千行级别），完全扛得住；真要上大数据量再做物化视图缓存。

## 你提过的问题，v2 这套设计怎么答

| 你的原问题 | v2 的答案 |
|---|---|
| 转账/借钱/押金要不要权责视图？ | 不要。只记 transaction，权责视图**默认不显示**；前端提供"全部视图"开关可切换到"仅展示不计入合计" |
| 女友代付要不要多角色？ | 不搞多角色。`counterparty="女友代付"` 或打标签 `#女友代付`，统计时按这个筛 |
| 工资 4500 里社保扣 450 怎么办？ | 两种都行：① 只记 income 4050；② 记 4500 income + 一条 450 的 EXPENSE（推荐，便于统计社保累计） |
| 10 号请假这天不给钱？ | 建一条 `type=ADJUST` 的 transaction 和一条 `source=ADJUST, amount=-150` 的 accrual_entry。动态算 +150 + 真实事件 -150 = 0 |
| 电动牙刷 15 天时 10/天，30 天时 5/天？ | `DYNAMIC_BY_DAY` 规则，**查询时**按"剩余成本 / 剩余预期天数"动态算 |
| 一箱苹果吃不完烂了？ | resource 状态改 `DISCARDED`，写一条 `source=END_SETTLE, amount=剩余成本` 把账平掉。你可以选择把损失归原分类（餐饮/零食），也可以归到新建的 `#损失` 标签下 |
| 可乐第一天三分之一，第二天三分之二？ | `BY_COUNT, total_qty=1`，两次打卡 qty 填 0.33 / 0.67 |
| 蛋白粉一三五七喝？ | `BY_COUNT`，喝的那天手动打卡；前端可提供"提醒"但不代为打卡 |
| 燕麦吃不固定？ | 同上，吃了打一下 |
| Steam 游戏爽完就不玩了？ | 两种：① 不管它，`ACTIVE` 状态让它继续动态摊；② 手动 `ENDED` + 一条 END_SETTLE 把剩余计入 |
| 键盘下月才到，中间时间算摊销吗？ | 看你：`start_use_at` 设成到货日；或 `include_start_gap=true` 把空窗也均摊 |
| 退货用了两天？ | 建 REFUND 交易，账户 IN 回款；权责**默认"只停未来"**（历史不动），也可选"全部回冲"写一条负值 ADJUST |
| 买菜单价/数量？ | transaction 的 `ext` JSON 里存 `{unit_price, qty, weight}` 等自定义字段，需要再取 |
| 想统计"一年吃了多少鸡蛋"？ | 标签打在 **消耗时的 accrual_entry.tags** 上才准，用 `JSON_CONTAINS(tags,'"鸡蛋"')` 求 SUM(qty)。买一袋鸡蛋时每次打卡打标签就行 |
| 消耗时要备注吗？ | `accrual_entry.note` 支持。想统计频次用标签（`#健身`/`#水煮`），想写感受用 note |
| 账户要不要分清楚（微信/支付宝/卡）？ | 默认就 1 个主账户，你懒就别分 |
| 像"育儿预算"、"彩礼金"怎么存？ | 建一个 account，账户就是"独立核算单元" |
| 分类能做到四级吗？ | 能。MVP 用代码常量，想要更深只需加一级 |
| 分类能共享/用户自定义吗？ | **MVP 不行**。系统级代码常量，未来 V2 再开放 |
| 评价/备注/购买链接/店铺？ | transaction 和 resource 都有 `note`（短备注）和 `ext` JSON（放链接、店铺、评分） |
| null 还是 JSON？ | 混合。会查询/建索引的字段正常列；规则/标签/扩展用 JSON。MySQL 8 的 JSON 够用 |
| 用户自定义统计？ | `ext` JSON 已预留；做 DSL 是 V2 目标，MVP 不做 |

## 核心概念（v2 版）

1. **transaction 就是单据，自己兼任现金流**
   v1 里 transaction 要派生 cash_entry，v2 砍掉了。单账户的交易直接用 transaction 读；转账靠 `to_account_id`。

2. **权责视图 = 动态计算 + 真实事件合并**
   按天摊销不入库，查询时当场算。真正入库的只有你手动做的动作（打卡、请假调整、丢弃结算）。

3. **标签直接 JSON 数组，别搞中间表**
   transaction 上一个 `tags`，accrual_entry 上一个 `tags`（默认继承，用户可改）。这两个标签语义不同：
   - transaction.tags = "这笔购买涉及什么"
   - accrual_entry.tags = "这次消耗具体是什么"
   - "一年吃了多少鸡蛋" 看后者更准。

4. **虚拟账户不要，用 counterparty 文本就够**
   借钱给小王、付房东押金、女友代付——这些关系不在你的账户表里，只在单条 transaction 的对手方字段里。统计时 GROUP BY counterparty 即可。

5. **分类/标签 MVP 走代码常量**
   表都不建（或只建只读表由代码 Seed）。用户选不自定义；需要自定义是 V2 的事。

## 动态计算权责视图的直观理解

假设 4-10 买了电动牙刷 150 元，规则 `DYNAMIC_BY_DAY, expected_days=30`。

**4-10 到 4-23（14 天）你查权责视图**：
- 服务端在内存里算：4-10 到 4-23 共 14 天，每天 5 元（因为一路均匀），合计 70 元
- 数据库里 accrual_entry 表里对这个 resource **0 条记录**（没打卡、没调整、没结算）
- 前端显示：4-10 ~ 4-23 每天 -5，共 -70
- 假设 4-15 这天你"不想记账了想休息"：在 4-15 记一条 `ADJUST -5`。前端显示：4-15 当天 5 - 5 = 0

**4-28 发现它坏了，丢了**：
- `resource.status=DISCARDED, end_at=2026-04-28`
- 补一条 `END_SETTLE, amount=150 - (已过天数×5 + 已有 ADJUST) = 损失金额`，归入原分类或专门 `#损失` 标签
- 4-29 起动态计算不再产出虚拟行（因为 status 不是 ACTIVE 且 end_at 已过）

**体验一致，数据库极简**。这就是 v2 的核心：能算的就不存。
