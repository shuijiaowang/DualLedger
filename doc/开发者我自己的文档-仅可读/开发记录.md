# DualLedger 开发记录（MVP 起步）

> 简记："改了什么 / 为什么这么改 / 坑 / 下一步"，一人一项目不追求 PR-style 细致。

## 2026-04-23 · MVP 第一版落地

### 做了什么

#### 后端（`service/`）
- 按 v2 数据库设计落地 5 张业务表 model：`Account / Resource / Transaction / AccrualEntry` + 保留 `User`；删除 `Example` model，不再进 AutoMigrate。
- 新增 `model/common.go`：自定义 `Money`（DECIMAL(14,2) 字符串 + `math/big` 运算，0 新依赖）、`JSONStrings / JSONMap / AmortizeRule` 三个 JSON 字段类型（自实现 `Scan/Value`）。
- 新增 `model/enum.go` 集中枚举 + 校验函数（`IsValidTxType / IsValidDirection / IsValidResourceStatus / IsValidAccrualSource / IsCashflowAccountingOnly / DefaultDirectionFor`）。
- 新增 `model/presets_category.go`（对齐文档 §5.2 清单） + `presets_tag.go` 建议词（不建表，走代码常量）。
- DAO 层：`dao/account.go / transaction.go / resource.go / accrual_entry.go`；`account.AdjustBalance` 走原子 SQL `balance + ?` 保证并发下也守恒，余额重算走 `SumBalanceFromTx`。
- Service 层：
  - `onboarding_service.go`：`EnsureDefaultAccount` 幂等，新用户注册事务内必建"主账户"。
  - `ledger_service.go`：`Create` 在同一事务里写 transaction + 同步账户余额 + 走 7 条校验（不变式 §15.1/§15.7）。
  - `resource_service.go`：创建 resource 时可一起写关联 EXPENSE/INCOME 交易；`Punch` 用"剩余成本/剩余数量"算金额保证闭合；`End` 对 BY_COUNT 自动写 END_SETTLE，FIXED_PERIOD/DYNAMIC_BY_DAY 默认不写（由动态行承担）。
  - `accrual_view_service.go`：核心动态计算。扫描 `ACTIVE+ENDED` 的 FIXED_PERIOD/DYNAMIC_BY_DAY 资源，对 [from,to) 区间每天生成虚拟行；合并真实 accrual_entry；可选把"即买即耗"的 transaction 一并展示。MVP 简化：DYNAMIC_BY_DAY 按 `total_cost/expected_days` 匀速均摊（稳定场景下与文档公式等价）。
  - `meta_service.go`：`/categories /tags /enums` 三端统一。
- API：补齐 `account / transaction / resource / accrual / meta` 五个模块；`routes/router.go` 把 meta 放到不需要 JWT 的 group，其他挂 JWT。
- 注册 hook：`service.UserService.Register` 改为事务封装，内部调用 `OnboardingService.EnsureDefaultAccount`，一次成功要么都成功。

### 约定/接口备忘
- 所有业务接口统一 `/api/*`，需要 JWT；`/api/user/login /register /api/meta/*` 无需 JWT。
- 响应格式延续老模板 `{ code, data, msg }`，code=0 成功，code=7 业务错误。
- 金额 API 层恒传字符串（`"15.00"`），前端 `normalizeAmount` 统一兜底。
- `transaction.amount` 恒 > 0；方向由 `type + direction`，不存负数（不变式 §15.1）。
- 软删除 transaction 不自动回退账户余额（避免隐式状态跳变），想回退用 `/accounts/:id/rebuild-balance`。

### 坑 / 取舍
1. **Go 依赖**：本想加 `shopspring/decimal`，但本机没 go 跑 `go get`，索性用 stdlib（big.Float）自己包一个 Money，零新依赖。代价：字符串外皮没编辑器补全友好。
2. **JSON 字段**：没加 `gorm.io/datatypes`，自己实现 `sql.Scanner/driver.Valuer`。好处是清晰，坏处是每加一种 JSON 类型都要复制一份模板。后续如果加更多字段，考虑抽个泛型 helper。
3. **DYNAMIC_BY_DAY 的简化**：文档原意"剩余成本/剩余预期天数"递推，MVP 直接 `total_cost/expected_days` 固定单价。稳定场景下结果一致；动态改 `expected_days` 时会有差异，等真实使用遇到再补回"递推版"。
4. **END_SETTLE 策略**：FIXED_PERIOD/DYNAMIC_BY_DAY 资源进终态时**不**写 END_SETTLE（由动态行已覆盖到 `end_at`），只有 BY_COUNT 写剩余成本为"损失"。避免与虚拟行重复计入。
5. **软删级联**：删除 transaction 会级联软删其衍生 accrual_entry（GORM transaction 内两条语句），但账户余额故意不回滚，理由同上。
6. **vitest.config 与 vite.config 不能 merge**：原模板 `vite.config` 导出的是 `({mode}) => config` callback，`mergeConfig` 拒绝合并，重写 vitest.config 为独立配置。
