# DualLedger 数据库设计

> 本文件是给 AI 协作/开发使用的权威规范。开发时请以本文件为准。
> 技术栈：MySQL 8.0（本地）/ 5.7（线上兼容）、Go + GORM、字符集 `utf8mb4_unicode_ci`。

---

## 0. v2 对 v1 的精简要点（开发者反馈驱动）

本版基于开发者对 v1 的审视意见做了大幅精简，核心意图是**"数据库只存真实事件，能算的就不存"**，避免冗余膨胀。

| 精简项 | v1（旧） | v2（新） | 理由 |
|---|---|---|---|
| **cash_entry 表** | 单独一张表承载所有现金流条目 | **删除**，用 transaction 自身 + `to_account_id` 表达 | 对于 INCOME/EXPENSE，cash_entry 与 transaction 字段几乎重复；TRANSFER 通过 `to_account_id` 即可 |
| **accrual_entry 的 AUTO 行** | 每日定时任务为 FIXED_PERIOD / DYNAMIC_BY_DAY 生成 1 行 | **不写行**，查询时按 resource 规则动态计算 | 避免会员/房租/工资每月爆出 30+ 行冗余数据；只存真实事件（手动打卡、调整、结算） |
| **transaction_tag / accrual_entry_tag 中间表** | 多对多中间表 | **删除**，改为业务行 `ext_json.tags`（或同语义 JSON 字段） | 开发者觉得中间表过于繁琐；JSON 数组足够表达，查询用 `JSON_CONTAINS` |
| **虚拟账户（is_virtual）** | 借出/押金/代付都开虚拟账户 | **不建**虚拟账户，transaction 加一个 `counterparty` 对手方文本字段 | 开发者反感"为了一次借款给小王建 id"；记录出入 + 对手方文本就够 |
| **resource.status = PAUSED** | 支持暂停 | **删除**，买了就一直算消耗 | 开发者认为"暂停 = 请假"，无必要；要彻底停就 ENDED |
| **amortize_rule.type = SCHEDULED** | 蛋白粉一三五七喝走计划表 | **删除**，改用 MANUAL 打卡 | 既然也要写 DB 行，不如让用户打卡 |
| **category / tag 的 user_id** | 每用户拷贝一份 | **删除**，改为系统级分类 + 标签 JSON | MVP 不让用户自定义分类，标签直接 JSON 记录 |
| **owner_user_id / payer_user_id** | 预留给代付多角色 | `payer_user_id` 删除；`owner_user_id` 仅在 category 预留且可空 | MVP 单用户，但保留分类升级位，后续放开 user 自定义更平滑 |
| **account.type / account.is_virtual** | 枚举 + 虚拟标记 | **删除**（保留 name + is_archived） | 账户就是独立钱袋，类型由用户 name 自表达 |
| **category.color / tag.color** | 颜色字段 | **删除** | 暂不考虑配色 |

**表数量**：v1 有 10 张表（user / account / category / tag / resource / transaction / cash_entry / accrual_entry / transaction_tag / accrual_entry_tag） → v2 精简到 **6 张业务表**（user / account / category / tag / resource / transaction / accrual_entry，其中 category / tag 可退化为代码常量）。

## 1. 设计原则（v2）

1. **双视图仍是核心**：一次业务 = 一条 `transaction` 主单 + 视情况 0~N 条 `accrual_entry`（权责真实事件）。"钱何时动"直接从 transaction 读；"价值何时耗"= 规则动态算 + 真实事件合并。
2. **能算的就不存**：规则性摊销（按天均摊、动态按天）不写行，查询时即时计算。只有"用户操作产生的真实事件"（打卡、请假调整、丢弃结算）才入 `accrual_entry`。
3. **固定列 + JSON 混合**：查询/索引用列；规则参数、标签、扩展字段（`amortize_rule`、`tags`、`ext`）用 JSON。
4. **软删除**：所有业务表都嵌入 `gorm.Model`。查询默认过滤软删除。
5. **单用户 MVP**：所有业务表带 `user_id` 做隔离，但不引入多人协作字段。"女友代付"之类场景用 `counterparty` 文本 + 单独分类表达。
6. **金额**：统一 `DECIMAL(14,2)`，单位元。禁用 float/double。
7. **时间**：统一 `DATETIME`。业务时间以 `*_at` 命名。
8. **命名**：下划线小写，表名单数，符合 GORM 默认。
9. **索引**：高频查询围绕 `(user_id, *_at)`；外键约束在 MVP 不加（应用层保证）。
10. **账户即钱袋**：`account` 是"一个独立核算单元"，不等于支付方式。新用户默认 1 个主账户，用户愿意细分再加。
11. **分类策略（兼容未来多用户）**：MVP 仅系统分类（静态初始化，不支持用户增删改）；为后续升级预留 `owner_user_id`（可空）+ `source=system/user`，不采用预留 id 段方案。
12. **标签策略（最终定稿）**：MVP 不建独立 tag 表、不做中间表；标签统一放在业务行 `ext_json.tags`（或同语义 JSON 字段）中，支持用户自定义语义（如 `egg_buy`、`egg_eat`）。
13. **标签统计口径**：先按标签过滤/匹配，再基于结构化字段（`qty/unit/amount`）统计；数量语义不写入标签后缀（避免 `鸡蛋_2` 这类解析复杂度）。

## 2. 核心模型关系（ER 概览）

```
user ──┬─< account                     （钱袋子 / 独立核算单元）
       ├─< resource                    （可摊销资源定义）
       └─< transaction                 （交易主单，自带 tags/cash 语义）
              │
              └─< accrual_entry        （权责真实事件：MANUAL/ADJUST/END_SETTLE）

resource ──< accrual_entry             （资源的每次真实消耗事件）

category（系统常量表，或直接代码枚举）
```

**业务不变式**：

- `transaction.amount > 0` 恒成立；方向由 `type` + `direction` 决定，不存负数。
- 关联了 `resource` 的交易：`(Σ accrual_entry.amount 真实事件) + (动态计算的累计摊销)` 在 resource 进入终态（ENDED/RETURNED/DISCARDED）时精确等于 `resource.total_cost`。
- 即买即耗（无 resource）的 EXPENSE：不写 `accrual_entry`，视图查询时将 transaction 本身作为"当日一次性权责事件"展示。
- `account.balance` 由 `transaction` 聚合重算；应用层每次写入 transaction 同步更新余额。

### 3.2 `account` — 资金账户（钱袋 / 独立核算单元）

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | 所属用户 |
| name | VARCHAR(64) | 否 | 显示名：如"主账户"、"银行卡微信支付宝"、"育儿基金" |
| balance | DECIMAL(14,2) | 否 | 当前余额（应用层维护的冗余字段；提供"按 transaction 重算"接口修数据） |
| currency | CHAR(3) | 否 | 默认 `CNY` |
| is_archived | TINYINT(1) | 否 | 1=已归档（从默认选择列表隐藏，历史记录保留） |

**索引**：`(user_id, is_archived, sort)`

**设计要点**：

- 账户名 = 用户心智。想叫"微信"就叫"微信"，想叫"女友的钱"就叫"女友的钱"，想叫"装修基金"就叫"装修基金"。
- `balance` 是冗余字段，真正来源 = 基于 transaction 求和。提供后台接口"rebuild balance for account X"。
- `is_archived` 语义 = **从录入界面隐藏**，不删除历史；不强制把该账户历史转移。

### 3.5 `resource` — 可摊销资源定义

> **何时建 resource？** 需要跨多天/多次消耗或按次数摊销的物品才建。一顿饭这种即买即耗不用建。

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| name | VARCHAR(128) | 否 | 电动牙刷 / 一箱苹果 / 年会员 / 2026 年 4 月工资 |
| category_code | VARCHAR(32) | 是 | 对应分类 code |
| unit | VARCHAR(16) | 是 | "天" / "个" / "次" / "ml" |
| total_qty | DECIMAL(14,4) | 是 | 总量（6 个、30 天、null） |
| remaining_qty | DECIMAL(14,4) | 是 | 剩余量（MANUAL/BY_COUNT 打卡时维护） |
| total_cost | DECIMAL(14,2) | 否 | 总成本（= 购买金额） |
| amortize_rule | JSON | 否 | 摊销规则，见 §4.2 |
| status | VARCHAR(16) | 否 | `ACTIVE` / `ENDED` / `RETURNED` / `DISCARDED` |
| purchase_at | DATETIME | 否 | 购买时间（= 出现金流那天） |
| start_use_at | DATETIME | 是 | 开始使用时间（摊销起点；默认 = `purchase_at`） |
| end_at | DATETIME | 是 | 结束时间（进入终态时写入） |
| note | VARCHAR(512) | 是 | |
| ext | JSON | 是 | 自定义字段（单价、标签、店铺、购买链接、评价、购买时长备注...） |

**`ResourceStatus` 枚举**（v2 **去掉 PAUSED**）：

- `ACTIVE` 使用中（默认）
- `ENDED` 正常结束（手动标记用完 / 达到 total_qty）
- `RETURNED` 退货
- `DISCARDED` 丢弃/烂了

**索引**：`(user_id, status, start_use_at)`；

### 3.6 `transaction` — 交易主单

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| type | VARCHAR(16) | 否 | 见 `TransactionType` |
| direction | VARCHAR(4) | 否 | `IN` / `OUT` / `BOTH`（`BOTH` 专用于 TRANSFER） |
| occur_at | DATETIME | 否 | 业务发生时间 |
| amount | DECIMAL(14,2) | 否 | 正数金额 |
| account_id | BIGINT UNSIGNED | 否 | 主账户（EXPENSE/INCOME/LOAN/DEPOSIT/REFUND/ADJUST 的账户；TRANSFER 的出方） |
| to_account_id | BIGINT UNSIGNED | 是 | 仅 TRANSFER 使用：入方账户 |
| category_code | VARCHAR(32) | 是 | 分类 code |
| resource_id | BIGINT UNSIGNED | 是 | 关联的资源（如买电动牙刷） |
| counterparty | VARCHAR(64) | 是 | 对手方文本（LOAN/DEPOSIT/REFUND 时用，如"小王"/"房东"/"女友"） |
| tags | JSON | 是 | 标签数组：`["鸡蛋","外卖"]` |
| title | VARCHAR(128) | 是 | 一句话描述 |
| note | VARCHAR(1024) | 是 | 长备注 |
| ext | JSON | 是 | 扩展字段（店铺、平台、评价、购买链接、`original_transaction_id` 等） |

**`TransactionType` 枚举**（v2 精简版，去 LOAN_REPAY 之类细分，用 direction 区分收付）：

- `INCOME` 收入（工资、红包）— direction 恒为 IN
- `EXPENSE` 支出（购物、吃饭）— direction 恒为 OUT
- `TRANSFER` 转账（银行卡↔微信、自有账户间移动）— direction 恒为 BOTH
- `LOAN` 借出/借入：OUT=借给别人，IN=别人借给我/别人还我
- `DEPOSIT` 押金：OUT=付押金，IN=退押金
- `REFUND` 退款/退货：IN=收到退款（`ext.original_transaction_id` 指向原 transaction）
- `ADJUST` 调整（请假扣款、余额修正等派生项）

**索引**：

- `(user_id, occur_at)` 主要查询路径
- `(user_id, type, occur_at)` 按类型过滤
- `(account_id, occur_at)` 账户流水
- `(to_account_id, occur_at)` 转账反查
- `(resource_id)` 反查资源的全部交易

### 3.7 `accrual_entry` — 权责真实事件条目

只存"用户真实操作"产生的条目

| 字段 | 类型 | 允许空 | 说明 |
|---|---|---|---|
| user_id | BIGINT UNSIGNED | 否 | |
| transaction_id | BIGINT UNSIGNED | 是 | 关联的源交易；手动打卡补录可为 NULL |
| resource_id | BIGINT UNSIGNED | 是 | 关联资源；不关联资源的一次性调整可为 NULL |
| category_code | VARCHAR(32) | 是 | 分类 code（默认继承自 transaction/resource，可覆盖） |
| amount | DECIMAL(14,2) | 否 | 金额；正数为消耗/产出，负数为冲减（请假、退货回冲） |

| note | VARCHAR(255) | 是 | |

**索引**：`(user_id, accrue_at)`、`(resource_id, accrue_at)`、`(transaction_id)`、`(category_code, accrue_at)`

## 4. 关键业务规则

### 4.1 双视图产出规则

| 场景 | transaction | accrual_entry |
|---|---|---|
| 吃饭 15 元（即买即耗） | 1 条 EXPENSE | **不写**（视图层把 transaction 当作当日权责展示） |
| 买电动牙刷 150 元 | 1 条 EXPENSE + resource_id | **不写**；视图层按 `amortize_rule` 动态计算 |
| 吃 1 个苹果（MANUAL 打卡） | 无 | 1 条 MANUAL, resource_id=一箱苹果 |
| 资源丢弃 / 烂了 → DISCARDED | 无 | 1 条 END_SETTLE, amount = `total_cost - 已真实累计` |
| 工资 4500 + 按 30 天均摊 | 1 条 INCOME + resource_id(工资包) | **不写**日条目；请假/迟到写 ADJUST 负值 |
| 请假 1 天 | 1 条 ADJUST（-150，account 可选不产生现金流） | 1 条 ADJUST, amount=-150, accrue_at=请假当天 |
| 转账 | 1 条 TRANSFER | 无 |
| 借出 200 给小王 | 1 条 LOAN, direction=OUT, counterparty="小王" | 无 |
| 退货（有 resource） | 1 条 REFUND + `ext.original_transaction_id` | 视策略：默认只停未来（不写负向）；可选回冲则写 1 条 ADJUST 负值 |

### 4.2 `amortize_rule` JSON Schema（3 种规则）

```json
// 类型 A：按固定天数平摊（会员、工资、房租）
{
  "type": "FIXED_PERIOD",
  "days": 30,
  "start": "2026-04-10",        // 缺省取 resource.start_use_at
  "include_start_gap": false     // 购买到使用之间的"空窗天"是否参与均摊
}

// 类型 B：按总次数/总量（一箱苹果 6 个、蛋白粉 30 次）
{
  "type": "BY_COUNT",
  "total_qty": 6
  // 使用阶段完全靠 MANUAL 打卡，不走规则自动推进
}

// 类型 C：动态按天（电动牙刷：每用一天，当前单日成本 = 剩余成本/剩余预期天数）
{
  "type": "DYNAMIC_BY_DAY",
  "expected_days": 30            // 预期寿命；用户可随时更新 //并且可为空，为空则是“今天”，需要判断今天大于购买时间，不为负。
}
```

### 4.3 权责视图的动态计算规则

权责视图展示某一时间区间的条目时，服务端/查询层**同时返回两类来源**：

1. **动态计算项**（不入库）：对用户所有 `status=ACTIVE/ENDED` 且 `amortize_rule.type ∈ {FIXED_PERIOD, DYNAMIC_BY_DAY}` 的 resource，按区间内每一天生成一条"虚拟权责条目"。
   - `FIXED_PERIOD`：每天金额 = `total_cost / days`；生效区间 `[max(start_use_at, start), min(start_use_at + days, end, resource.end_at)]`
   - `DYNAMIC_BY_DAY`：按"当日剩余成本 / 当日剩余预期天数"滚动计算；若 resource 已 ENDED 则停止在 end_at。
2. **真实事件条目**（入库）：对区间内的 `accrual_entry` 做标准查询。

两类合并后做分类/标签聚合。前端感知不到区别；统计准确闭合：动态项 Σ + 真实事件 Σ = 资源总成本（在终态时）。

**实现要点**：

- 动态计算产出是**只读虚拟行**，不能被用户编辑；要表达"某天不摊销"走 `ADJUST` 真实事件条目（累加即可抵消）。
- 性能：动态计算 O(天数 × ACTIVE 资源数)，在 10^3 量级内足够；极端情况可加缓存或按月预聚合物化视图。
- 幂等：因为不写行，不存在重复生成问题。
- 切换时区/跨月查询：以 `accrue_at` 的日期组件为聚合键。

### 4.4 账户余额一致性

- 应用层每次写入 transaction 时同步更新 `account.balance`（事务内）：
  - INCOME/LOAN-IN/DEPOSIT-IN/REFUND：`account_id.balance += amount`
  - EXPENSE/LOAN-OUT/DEPOSIT-OUT/ADJUST（负向）：`account_id.balance -= amount`
  - TRANSFER：`account_id.balance -= amount` 且 `to_account_id.balance += amount`
- 提供后台接口"根据 transaction 重算账户余额"用于修数据。

### 4.5 转账/借钱/押金/代付

| 场景 | 记法 |
|---|---|
| 银行卡→微信 100 | `type=TRANSFER, direction=BOTH, account_id=招行, to_account_id=微信, amount=100` |
| 女友给我 50 帮买外卖（我不关心谁的） | 用户创建新账户(女友账户) |
| 借出 200 给小王 | `type=LOAN, direction=OUT, amount=200, account_id=微信, counterparty="小王"` |
| 小王还我 200 | `type=LOAN, direction=IN, amount=200, account_id=微信, counterparty="小王"` |
| 付房东押金 500 | `type=DEPOSIT, direction=OUT, amount=500, account_id=招行, counterparty="房东"` |
| 退还押金 500 | `type=DEPOSIT, direction=IN, amount=500, account_id=招行, counterparty="房东"` |

**暂定权责视图对以上类型均不展示**（纯资金运输，不参与价值消耗）。前端可以提供"全部视图"开关以"仅展示不计入合计"的形式显示。

### 4.7 工资与请假的最小方案

1. 发薪时建 `resource`：name="2026-04 工资"，total_cost=4500，amortize_rule=`{type:FIXED_PERIOD,days:30,start:"2026-04-01"}`，status=ACTIVE。
2. 建 `transaction: type=INCOME, amount=4500, account_id=招行, resource_id=工资id`，账户余额 +4500。
3. **不需要每日定时任务**：视图按规则动态算出每天 +150。
4. 10 号请假：建 `transaction: type=ADJUST, amount=150, direction=OUT（或 note 里说明），resource_id=工资id` + `accrual_entry: source=ADJUST, amount=-150, accrue_at=2026-04-10, resource_id=工资id`。视图当日合计 = 150（动态）- 150（ADJUST）= 0。
5. 15 号迟到扣 20：同上，amount=-20。
6. 社保扣 450：若发薪前扣，income.amount 直接 4050；若独立扣款，单建 EXPENSE。也可以用户单独分开记录。

## 9. 典型查询示例

```sql
-- 1. 某月现金流（按日聚合）：直接从 transaction 算
--    EXPENSE/LOAN-OUT/DEPOSIT-OUT/ADJUST 减，INCOME/LOAN-IN/DEPOSIT-IN/REFUND 加，TRANSFER 两侧各展开
SELECT DATE(occur_at) d,
       SUM(
         CASE
           WHEN type='TRANSFER' THEN 0                              -- 忽略（净流为 0）
           WHEN direction='IN'  THEN amount
           WHEN direction='OUT' THEN -amount
         END
       ) net
FROM transaction
WHERE user_id=? AND occur_at >= ? AND occur_at < ?
  AND type IN ('INCOME','EXPENSE','REFUND','ADJUST')
  AND deleted_at IS NULL
GROUP BY d;

-- 2. 某账户流水（包含转账对侧）
SELECT * FROM transaction
WHERE user_id=? AND deleted_at IS NULL
  AND (account_id=:aid OR to_account_id=:aid)
  AND occur_at >= ? AND occur_at < ?
ORDER BY occur_at DESC;

-- 3. 真实权责事件（不含动态摊销；动态部分由服务端代码合并）
SELECT category_code, SUM(amount)
FROM accrual_entry
WHERE user_id=? AND accrue_at >= ? AND accrue_at < ?
  AND deleted_at IS NULL
GROUP BY category_code;

-- 4. 查询按规则类型过滤的 ACTIVE 资源（供动态计算引擎扫描）
SELECT * FROM resource
WHERE user_id=? AND status='ACTIVE'
  AND JSON_EXTRACT(amortize_rule, '$.type') IN ('FIXED_PERIOD','DYNAMIC_BY_DAY');

-- 5. 一年吃了多少鸡蛋（消耗视角 · 用 accrual_entry.tags JSON）
SELECT SUM(qty) AS total_eggs
FROM accrual_entry
WHERE user_id=?
  AND JSON_CONTAINS(tags, '"鸡蛋"')
  AND accrue_at BETWEEN ? AND ?
  AND deleted_at IS NULL;

-- 6. 一年"花了多少钱跟鸡蛋有关"（购买视角 · 偏大，仅参考）
SELECT SUM(amount) FROM transaction
WHERE user_id=?
  AND JSON_CONTAINS(tags, '"鸡蛋"')
  AND occur_at BETWEEN ? AND ?
  AND type='EXPENSE'
  AND deleted_at IS NULL;

-- 7. 小王欠我多少
SELECT SUM(CASE direction WHEN 'OUT' THEN amount ELSE -amount END) AS owed_by_peer
FROM transaction
WHERE user_id=? AND type='LOAN' AND counterparty='小王' AND deleted_at IS NULL;
```
